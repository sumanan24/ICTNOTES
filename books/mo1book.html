1. Introduction

Welcome to the User Requirements for Software Solution module. This comprehensive reading book is designed to guide you through the essential processes of understanding business requirements, gathering user needs, analyzing requirements, and conducting feasibility studies for software solutions.

This module provides a structured approach to requirement engineering, covering the complete lifecycle from initial planning through to feasibility assessment. You will learn industry-standard techniques and methodologies that are widely used in software development projects.

The content is organized into five main elements, each building upon the previous one to provide you with a complete understanding of the requirement gathering and analysis process. Through practical examples, case studies, and hands-on exercises, you will develop the skills necessary to work effectively as a requirements analyst or business analyst in the software industry.

How to Use This Book:
  • Read each section sequentially to build your understanding
  • Complete the practical sessions to reinforce learning
  • Review the performance criteria to ensure you meet learning objectives
  • Use the examples and case studies as reference materials

2. Element 1: Introduction to Various Business Processes

This element focuses on understanding business processes and planning the initial scope of software solutions. You will learn to identify solution ranges, work with stakeholders, and plan schedules for requirement gathering activities.

2.1 Identify and Outline Range of Proposed Solution

Learning Objectives

By the end of this session, you will be able to:

  • Identify contractual document scope
  • Outline solution range within defined scope
  • Apply organizational standards to document solutions

Understanding Contractual Documents and Scope

Before beginning any software development project, it is essential to understand the contractual agreement between the client and the development organization. The contractual document defines the boundaries, expectations, and deliverables of the project.

Key Components of Contractual Documents:
  • Project Scope: Defines what will and will not be included in the project
  • Deliverables: Specific outputs expected from the project
  • Timeline: Project schedule and milestones
  • Budget: Financial constraints and payment terms
  • Standards: Quality and documentation standards to be followed

Identifying Scope Elements

When reviewing a contractual document, you should identify:

  1. In-scope items: Features, functions, and deliverables explicitly included
  2. Out-of-scope items: Items explicitly excluded from the project
  3. Assumptions: Conditions taken for granted
  4. Constraints: Limitations that affect the project

2.1 Identify and Outline Range of Proposed Solution (Continued)

Example: A contract for a library management system might specify:
In-scope: Book cataloging, member registration, loan management
Out-of-scope: Online payment integration, mobile application
Assumptions: Library has existing network infrastructure
Constraints: Must comply with data protection regulations

Organizational Standards

Organizations typically have standards and guidelines for documenting solutions. These may include:

  • Documentation templates and formats
  • Naming conventions
  • Version control procedures
  • Review and approval processes

Creating Solution Range Outlines

When outlining the solution range, you should:

  1. Review the contractual document thoroughly
  2. Extract all scope-related information
  3. Organize the information according to organizational standards
  4. Create a clear, structured outline document
  5. Have the outline reviewed and approved by relevant stakeholders
Performance Criteria PC 1.1: Range of the proposed solution within the scope defined in the contractual document identified and outlined as per the organizational standards.

2.2 Identify and Agree Upon Key Stakeholders

Learning Objectives

By the end of this session, you will be able to:

  • Identify stakeholders from different functional areas
  • Explain stakeholder roles and involvement
  • Agree upon stakeholders with clients

Understanding Stakeholders

Stakeholders are individuals or groups who have an interest in or are affected by the software solution. Identifying the right stakeholders is crucial for successful requirement gathering.

Types of Stakeholders

Stakeholder Type Description Example Roles
Primary Stakeholders Directly affected by the system End users, System administrators
Secondary Stakeholders Indirectly affected or have influence Managers, IT support staff
Key Decision Makers Have authority to approve or reject Project sponsors, Department heads
Technical Stakeholders Involved in technical aspects Developers, System architects

Stakeholder Identification Process

The process of identifying stakeholders involves:

  1. Brainstorming: List all potential stakeholders
  2. Categorization: Group stakeholders by functional area
  3. Prioritization: Identify key stakeholders
  4. Verification: Confirm with client organization
  5. Documentation: Create stakeholder register
Stakeholder Identification Process
Brainstorm → Categorize → Prioritize → Verify → Document

Functional Areas

Stakeholders typically come from various functional areas:

  • Operations: Day-to-day users of the system
  • Management: Decision makers and supervisors
  • IT Department: Technical support and infrastructure
  • Finance: Budget and cost management
  • Legal/Compliance: Regulatory and legal requirements
  • Customer Service: External customer interaction

Stakeholder Agreement

Once stakeholders are identified, it is important to:

  • Present the stakeholder list to the client
  • Explain the role and importance of each stakeholder
  • Obtain formal agreement on the stakeholder list
  • Document the agreement in writing
  • Establish communication channels with each stakeholder
Example Stakeholder Register:
Name Role Functional Area Contact
John Smith Operations Manager Operations john.smith@company.com
Sarah Johnson IT Director IT Department sarah.j@company.com
Performance Criteria PC 1.2: Key stakeholders from respective functional areas identified and agreed upon with the client.

2.3 Plan Time Schedule and Arrange Meetings

Learning Objectives

By the end of this session, you will be able to:

  • Identify time requirements for requirement gathering activities
  • Create time schedules for requirement gathering process
  • Arrange meetings with key stakeholders

Planning Requirement Gathering Activities

Effective planning is essential for successful requirement gathering. You need to estimate the time required for each activity and create a realistic schedule.

Requirement Gathering Activities

Common activities include:

  • Interviews: One-on-one sessions with stakeholders (1-2 hours each)
  • Workshops: Group sessions for collaborative requirement gathering (2-4 hours)
  • Document Review: Analyzing existing documentation (varies by volume)
  • Observation: Watching current processes in action (varies)
  • Questionnaires: Distributed surveys (1-2 weeks for responses)

Creating Time Schedules

When creating schedules, consider:

  1. Activity Duration: How long each activity will take
  2. Dependencies: Activities that must be completed before others
  3. Resource Availability: When stakeholders are available
  4. Buffer Time: Extra time for unexpected delays
  5. Review and Approval: Time for document review and sign-off
Example Schedule:
Week Activity Duration
Week 1 Stakeholder identification and agreement 2 days
Week 1-2 Document review 5 days
Week 2-3 Stakeholder interviews 10 days
Week 3 Workshop sessions 3 days
Week 4 Observation and process analysis 5 days

Arranging Meetings

When arranging meetings with stakeholders:

  • Send meeting invitations well in advance (at least 1 week)
  • Provide clear agenda and objectives
  • Confirm availability and send reminders
  • Prepare necessary materials and questions
  • Schedule follow-up meetings if needed
  • Document meeting outcomes and action items
Important: Always confirm meeting times with stakeholders and send calendar invitations. Be respectful of stakeholders' time and prepare thoroughly for each meeting.
Performance Criteria PC 1.3: Time schedule for the requirement gathering process planned and conveyed to the respective key stakeholders and meetings arranged accordingly.

3. Study of Business Processes

This element focuses on understanding existing business processes and gathering requirements through various techniques. You will learn to use data gathering methods, observe workflows, identify source documents, and record key decisions.

3.1 Identify Requirements through Data Gathering Techniques

Learning Objectives

By the end of this session, you will be able to:

  • Identify various data gathering techniques
  • Select appropriate techniques for different situations
  • Gather requirements from users using selected techniques

Data Gathering Techniques

There are several techniques available for gathering requirements from users. Each technique has its strengths and weaknesses, and the choice depends on the situation, stakeholders, and type of information needed.

1. Document Review

Description: Analyzing existing documentation such as business process documents, user manuals, system documentation, and reports.

When to use: When you need to understand existing systems, processes, or historical information.

Advantages:

  • Provides historical context
  • Less time-consuming than interviews
  • Can reveal implicit requirements

Disadvantages:

  • Documents may be outdated
  • May not reflect current practices
  • Requires careful analysis

2. Interviews

Description: One-on-one or small group conversations with stakeholders to gather detailed information.

Types:

  • Structured: Predefined questions
  • Unstructured: Open-ended conversation
  • Semi-structured: Combination of both

When to use: For detailed, in-depth information from key stakeholders.

Advantages:

  • Allows for detailed exploration
  • Can clarify ambiguities immediately
  • Builds rapport with stakeholders

Disadvantages:

  • Time-consuming
  • May be influenced by interviewer bias
  • Requires good communication skills
Interview Best Practices:
  • Prepare questions in advance
  • Create a comfortable environment
  • Listen actively and take notes
  • Ask follow-up questions
  • Summarize key points at the end

3. Questionnaires

Description: Written surveys distributed to multiple stakeholders to gather standardized information.

When to use: When you need information from many stakeholders or when time is limited.

Advantages:

  • Can reach many people efficiently
  • Standardized responses
  • Less time-consuming than interviews
  • Can be anonymous

Disadvantages:

  • Low response rates
  • Limited opportunity for clarification
  • May miss important details

4. Workshops

Description: Facilitated group sessions where stakeholders collaborate to identify and discuss requirements.

When to use: When you need to build consensus or gather input from multiple stakeholders simultaneously.

Advantages:

  • Encourages collaboration
  • Builds consensus
  • Can generate creative solutions
  • Efficient use of stakeholder time

Disadvantages:

  • Requires skilled facilitation
  • May be dominated by vocal participants
  • Can be difficult to schedule

5. Observations

Description: Watching users perform their work in their natural environment.

Types:

  • Participant: Observer participates in the process
  • Non-participant: Observer watches without participating

When to use: When you need to understand actual work practices that may differ from documented procedures.

Advantages:

  • Reveals actual work practices
  • Can identify inefficiencies
  • Provides context for requirements

Disadvantages:

  • Time-consuming
  • May alter behavior (Hawthorne effect)
  • Requires permission and access

6. Prototypes

Description: Creating early versions or mockups of the system to gather feedback.

Types:

  • Low-fidelity: Paper sketches, wireframes
  • High-fidelity: Interactive mockups, working prototypes

When to use: When requirements are unclear or when you need early user feedback.

Advantages:

  • Helps users visualize the solution
  • Encourages early feedback
  • Can reveal unstated requirements

Disadvantages:

  • Can be time-consuming to create
  • May set unrealistic expectations
  • Requires technical skills
Data Gathering Techniques Comparison
Document Review → Interviews → Questionnaires → Workshops → Observations → Prototypes
(Low interaction) → (High interaction) → (Medium interaction)

Selecting Appropriate Techniques

The choice of technique depends on several factors:

Factor Consideration
Number of stakeholders Questionnaires for many, interviews for few
Type of information Interviews for detailed, questionnaires for standardized
Time available Questionnaires faster, observations slower
Stakeholder availability Workshops require coordination, document review is flexible
Clarity of requirements Prototypes when unclear, interviews when clear
Performance Criteria PC 2.1: Requirements identified through data gathering techniques (document review, interview, questionnaires, workshops, observations, prototypes) with respective users.

3.2 Recognize Present Workflow by Observing Existing Processes

Learning Objectives

By the end of this session, you will be able to:

  • Identify observation techniques for process analysis
  • Explain how to recognize and document workflows
  • Apply observation skills to analyze existing processes

Understanding Workflows

A workflow is a sequence of tasks or activities that transform inputs into outputs. Understanding existing workflows is essential for identifying requirements and potential improvements.

Workflow Components

  • Inputs: Resources, data, or materials that enter the process
  • Activities: Tasks or steps performed
  • Outputs: Results or deliverables produced
  • Actors: People or systems performing activities
  • Decisions: Points where choices are made
  • Flow: Sequence and relationships between activities

Observation Techniques

When observing processes, you can use:

  1. Direct Observation: Watch the process as it happens
  2. Shadowing: Follow a user through their workday
  3. Process Mapping: Document the flow of activities
  4. Time Studies: Measure how long activities take
  5. Video Recording: Record for later analysis (with permission)

Documenting Workflows

Workflows can be documented using:

  • Flowcharts: Visual representation of process flow
  • Swimlane Diagrams: Show responsibilities by role
  • Process Maps: High-level overview of processes
  • Narrative Descriptions: Written descriptions of processes
Example Workflow - Order Processing:
1. Customer submits order → 2. Order received → 3. Check inventory →
4. [If available] Process payment → 5. Prepare shipment → 6. Ship order →
[If not available] Notify customer → Wait for stock
Performance Criteria PC 2.2: Present workflow recognized by observing the existing processes.

3.3 Identify Source Documents from Present System

Learning Objectives

By the end of this session, you will be able to:

  • Identify types of source documents in existing systems
  • Explain how to extract information from source documents
  • Apply organizational standards to document identification

Source Documents

Source documents are original records that provide evidence of business transactions or activities. They are essential for understanding data requirements and business rules.

Types of Source Documents

Document Type Purpose Examples
Transaction Documents Record business transactions Invoices, receipts, purchase orders
Reference Documents Provide reference information Price lists, catalogs, manuals
Authorization Documents Provide approval or authorization Approval forms, authorization letters
Reporting Documents Summarize information Reports, statements, summaries

Identifying Source Documents

To identify source documents:

  1. Review existing system documentation
  2. Observe processes and note documents used
  3. Interview users about documents they work with
  4. Review file systems and document storage
  5. Analyze data entry points in current systems
Performance Criteria PC 2.3: Source documents from the present system identified as per the organization standards.

3.4 Record Key Decisions from Client Meetings

Learning Objectives

By the end of this session, you will be able to:

  • Identify and recall SOP requirements for meeting documentation
  • Explain how to identify and record key decisions
  • Apply SOP to analyze meeting scenarios and create decision documentation

Standard Operating Procedures (SOP)

Organizations typically have Standard Operating Procedures (SOP) for documenting meetings and decisions. These procedures ensure consistency and completeness in documentation.

Key Components of Meeting Documentation

  • Meeting Information: Date, time, location, attendees
  • Agenda Items: Topics discussed
  • Decisions Made: Key decisions and outcomes
  • Action Items: Tasks assigned with owners and deadlines
  • Next Steps: Follow-up activities

Recording Key Decisions

When recording decisions, include:

  1. Decision Statement: Clear description of what was decided
  2. Rationale: Reasons for the decision
  3. Decision Maker: Who made the decision
  4. Date: When the decision was made
  5. Impact: How this decision affects the project
Example Decision Record:
Decision: Use cloud-based hosting for the new system
Rationale: Provides scalability and reduces infrastructure costs
Decision Maker: IT Director and Project Sponsor
Date: 2024-01-15
Impact: Requires cloud security assessment and vendor selection
Performance Criteria PC 2.4: Key decisions of the client meetings recorded according to the SOP (Standard Operating Procedures).

4. Requirements Gathering

This element focuses on documenting and organizing requirements. You will learn to create requirement specifications, distinguish between functional and non-functional requirements, and suggest business process improvements.

4.1 Document Requirement Specifications

Learning Objectives

By the end of this session, you will be able to:

  • Identify organization standards and guidelines for documentation
  • Explain requirement specification documentation structure
  • Apply standards to create comprehensive requirement specifications

Requirement Specification Documents

A Requirement Specification Document (RSD) is a formal document that describes the functional and non-functional requirements of a software system. It serves as a contract between stakeholders and developers.

Document Structure

Typical sections of an RSD include:

  1. Introduction: Purpose, scope, definitions, references
  2. Overall Description: Product perspective, functions, user characteristics
  3. System Features: Detailed functional requirements
  4. External Interface Requirements: User, hardware, software, communication interfaces
  5. Non-functional Requirements: Performance, security, usability, etc.
  6. Appendices: Glossary, diagrams, references

Writing Good Requirements

Requirements should be:

  • Complete: All necessary information included
  • Clear: Unambiguous and easy to understand
  • Consistent: No contradictions
  • Verifiable: Can be tested or validated
  • Traceable: Linked to source and design
Important: Always follow organizational standards when creating requirement documents. These standards ensure consistency and facilitate review and approval processes.
Performance Criteria PC 3.1: Requirement specifications documented as per the organization standards and guidelines.

4.2 Highlight Functional and Non-functional Requirements

Learning Objectives

By the end of this session, you will be able to:

  • Define functional and non-functional requirements
  • Explain how to highlight and categorize requirements
  • Apply standards to document functional and non-functional requirements

Functional Requirements

Functional requirements describe what the system should do - the functions, features, and behaviors it must provide.

Characteristics of Functional Requirements

  • Describe system behavior and functions
  • Specify inputs and expected outputs
  • Define business rules and logic
  • Can be verified through testing
Example Functional Requirements:
  • The system shall allow users to create new accounts
  • The system shall validate user login credentials
  • The system shall generate monthly reports
  • The system shall process payment transactions

Non-functional Requirements

Non-functional requirements describe how well the system should perform - quality attributes and constraints.

Types of Non-functional Requirements

Type Description Examples
Performance Speed and efficiency requirements Response time, throughput, resource usage
Security Protection of data and systems Authentication, encryption, access control
Usability Ease of use and user experience User interface design, learnability
Reliability System availability and fault tolerance Uptime, error recovery, backup
Scalability Ability to handle growth User capacity, data volume
Maintainability Ease of maintenance and updates Code quality, documentation
Example Non-functional Requirements:
  • The system shall respond to user requests within 2 seconds
  • The system shall support 1000 concurrent users
  • The system shall encrypt all sensitive data
  • The system shall be available 99.9% of the time

Categorizing Requirements

When documenting requirements, clearly separate and label:

  • Functional requirements (FR)
  • Non-functional requirements (NFR)
  • Business requirements (BR)
  • User requirements (UR)
  • System requirements (SR)
Requirements Hierarchy
Business Requirements
↓
User Requirements
↓
System Requirements
├─ Functional Requirements
└─ Non-functional Requirements
Performance Criteria PC 3.2: Functional and non-functional requirements of the proposed solutions highlighted, as per the organization standards and guidelines.

4.3 Include Suggested Business Process Improvements

Learning Objectives

By the end of this session, you will be able to:

  • Identify business process improvement opportunities
  • Explain how to suggest and document improvements
  • Apply knowledge to create improvement recommendations

Business Process Improvement

While gathering requirements, you may identify opportunities to improve existing business processes. These improvements should be documented and presented to stakeholders.

Identifying Improvement Opportunities

Look for:

  • Bottlenecks: Processes that slow down workflow
  • Redundancies: Duplicate or unnecessary steps
  • Manual Processes: Tasks that could be automated
  • Error-prone Steps: Areas with frequent mistakes
  • Inefficiencies: Processes that waste time or resources

Documenting Improvements

When documenting improvements, include:

  1. Current State: Description of current process
  2. Issues: Problems with current process
  3. Proposed Improvement: Suggested change
  4. Benefits: Expected advantages
  5. Implementation Considerations: What needs to be done
Example Process Improvement:
Current State: Orders are manually entered into the system by staff
Issue: High error rate, time-consuming
Proposed Improvement: Implement online order form with validation
Benefits: Reduced errors, faster processing, 24/7 availability
Implementation: Develop web form, integrate with existing system
Performance Criteria PC 3.3: Suggested business process improvements included as per the requirements.

5. Requirement Analysis

This element focuses on presenting requirements to clients, developing prototypes, and obtaining confirmation. You will learn effective presentation techniques and prototype development methods.

5.1 Present and Explain Requirement Specifications to Client

Learning Objectives

By the end of this session, you will be able to:

  • Identify presentation techniques and organization standards
  • Explain how to present and explain requirements effectively
  • Apply presentation skills to communicate requirements to clients

Presenting Requirements

Presenting requirements effectively is crucial for ensuring client understanding and obtaining approval. The presentation should be clear, organized, and tailored to the audience.

Presentation Preparation

Before presenting:

  1. Know Your Audience: Understand stakeholders' backgrounds and interests
  2. Prepare Materials: Create slides, handouts, and visual aids
  3. Practice: Rehearse your presentation
  4. Anticipate Questions: Prepare answers for likely questions
  5. Check Technology: Ensure equipment works properly

Presentation Structure

A typical presentation includes:

  1. Introduction: Overview and objectives
  2. Background: Context and scope
  3. Requirements Overview: High-level summary
  4. Detailed Requirements: Key functional and non-functional requirements
  5. Next Steps: Timeline and action items
  6. Q&A: Address questions and concerns

Effective Communication Techniques

  • Use Visual Aids: Diagrams, charts, and graphs
  • Speak Clearly: Use simple language, avoid jargon
  • Engage Audience: Ask questions, encourage participation
  • Provide Examples: Use real-world scenarios
  • Summarize Key Points: Reinforce important information
Presentation Tips:
  • Start with the big picture, then provide details
  • Use visual aids to illustrate complex concepts
  • Allow time for questions throughout
  • Be prepared to adjust based on audience feedback
  • Follow up with written documentation
Performance Criteria PC 4.1: Requirement specifications, presented and explained to the client as per the organization standards.

5.2 Develop and Present Prototypes

Learning Objectives

By the end of this session, you will be able to:

  • Identify prototyping techniques and tools
  • Explain prototype development process
  • Apply prototyping skills to create and present solution prototypes

Prototyping

Prototyping involves creating early versions or mockups of the system to demonstrate functionality and gather feedback. Prototypes help stakeholders visualize the solution and identify issues early.

Types of Prototypes

Type Description When to Use
Low-fidelity Simple sketches, wireframes, paper prototypes Early stages, quick iterations
High-fidelity Detailed, interactive mockups When detailed feedback needed
Horizontal Shows many features at surface level To show breadth of functionality
Vertical Shows few features in depth To demonstrate specific functionality
Throwaway Discarded after feedback For exploration and learning
Evolutionary Refined into final product When prototype will become product

Prototyping Tools

Common tools for creating prototypes include:

  • Paper/Sketching: Quick, low-cost, easy to modify
  • Wireframing Tools: Balsamiq, Figma, Adobe XD
  • UI Design Tools: Sketch, InVision, Axure
  • Development Tools: HTML/CSS/JS for interactive prototypes

Prototype Development Process

  1. Identify Key Features: Determine what to prototype
  2. Choose Fidelity Level: Low or high fidelity
  3. Create Prototype: Build using selected tools
  4. Test with Users: Gather feedback
  5. Iterate: Refine based on feedback
  6. Present to Stakeholders: Demonstrate functionality
Example Prototype Scenario:
For a library management system, create a low-fidelity prototype showing:
- Login screen layout
- Book search interface
- Checkout process flow
- User dashboard layout
Performance Criteria PC 4.2: Prototypes developed and presented to the client as per the organization standards.

5.3 Obtain Client Confirmation for Requirement Specification

Learning Objectives

By the end of this session, you will be able to:

  • Identify confirmation processes and documentation requirements
  • Explain how to obtain formal client confirmation
  • Apply processes to secure requirement specification approval

Obtaining Client Confirmation

Client confirmation is a formal approval process where stakeholders review and approve the requirement specification document. This approval serves as a baseline for development.

Confirmation Process

  1. Document Review: Provide complete requirement specification to client
  2. Presentation: Present and explain requirements
  3. Feedback Collection: Gather comments and concerns
  4. Revision: Update document based on feedback
  5. Final Review: Present revised document
  6. Formal Approval: Obtain signed confirmation

Confirmation Documentation

Confirmation typically includes:

  • Approval Form: Formal document with signatures
  • Version Control: Document version number and date
  • Approver Information: Name, role, date of approval
  • Conditions: Any conditions or reservations
  • Change Control: Process for future changes
Important: Never proceed with development without formal client confirmation. This protects both parties and ensures alignment on requirements.
Performance Criteria PC 4.3: Client confirmation for requirement specification obtained as per the organization standards.

6. Feasibility Study

This element focuses on evaluating the feasibility of proposed software solutions. You will learn to assess technical, economic, operational, and schedule feasibility.

6.1 Conduct Feasibility Study for Software Solutions

Learning Objectives

By the end of this session, you will be able to:

  • Identify types of feasibility studies
  • Explain how to conduct feasibility assessments
  • Apply knowledge to evaluate software solution feasibility

Feasibility Study Overview

A feasibility study evaluates whether a proposed software solution is viable and worth pursuing. It helps stakeholders make informed decisions before committing significant resources.

Types of Feasibility

Type Description Key Questions
Technical Feasibility Can the solution be built with available technology? Is technology available? Do we have required skills? Are there technical constraints?
Economic Feasibility Is the solution financially viable? What are the costs? What are the benefits? What is the ROI?
Operational Feasibility Can the solution be operated effectively? Will users accept it? Can it be integrated? Are resources available?
Schedule Feasibility Can the solution be delivered on time? Is timeline realistic? Are resources available? Are there dependencies?
Legal Feasibility Does the solution comply with laws and regulations? Are there legal restrictions? Does it comply with regulations? Are there licensing issues?

Conducting Technical Feasibility

Technical feasibility assesses:

  • Availability of required technology
  • Technical skills and expertise of team
  • Hardware and software requirements
  • Integration with existing systems
  • Scalability and performance requirements

Conducting Economic Feasibility

Economic feasibility includes:

  • Cost Analysis: Development, implementation, maintenance costs
  • Benefit Analysis: Direct and indirect benefits
  • Cost-Benefit Analysis: Comparison of costs and benefits
  • Return on Investment (ROI): Financial return calculation
  • Payback Period: Time to recover investment
Example Economic Feasibility:
Costs: Development: $100,000, Implementation: $20,000, Annual Maintenance: $10,000
Benefits: Annual savings: $30,000, Increased revenue: $15,000/year
ROI: (45,000 - 10,000) / 120,000 = 29% annually
Payback Period: 120,000 / 35,000 = 3.4 years

Conducting Operational Feasibility

Operational feasibility evaluates:

  • User acceptance and willingness to change
  • Organizational readiness
  • Training requirements
  • Impact on existing processes
  • Resource availability for operations

Conducting Schedule Feasibility

Schedule feasibility considers:

  • Project timeline and deadlines
  • Resource availability
  • Dependencies between activities
  • Risk factors that could cause delays
  • Buffer time for unexpected issues

Feasibility Study Report Structure

  1. Executive Summary: Overview and recommendation
  2. Introduction: Background and objectives
  3. Current System: Description of existing situation
  4. Proposed Solution: Description of proposed system
  5. Feasibility Analysis: Technical, economic, operational, schedule
  6. Risks and Mitigation: Identified risks and solutions
  7. Recommendation: Go/no-go decision with rationale
  8. Appendices: Supporting documents and data
Feasibility Study Best Practices:
  • Be objective and unbiased
  • Consider all types of feasibility
  • Use quantitative data when possible
  • Identify and address risks
  • Provide clear recommendations
  • Document assumptions
Performance Criteria PC 5.1: Feasibility study for software solutions conducted as per the organization standards.

6.2 Practical Sessions

In this practical session, you will:

  1. Conduct a technical feasibility assessment for a given scenario
  2. Perform cost-benefit analysis
  3. Evaluate operational feasibility
  4. Create a feasibility study report
  5. Present findings and recommendations

7. Practical Assignments

This section provides practical assignments to reinforce learning and develop practical skills in requirement gathering and analysis.

Assignment 1: Stakeholder Identification and Planning

Scenario: A local library wants to implement a new library management system.

Tasks:

  1. Identify key stakeholders from different functional areas
  2. Create a stakeholder register
  3. Develop a time schedule for requirement gathering
  4. Plan meetings with identified stakeholders

Deliverables: Stakeholder register, time schedule, meeting plan

Assignment 2: Requirements Gathering

Scenario: Continue with the library management system project.

Tasks:

  1. Conduct interviews with at least 3 stakeholders
  2. Observe existing library processes
  3. Identify source documents
  4. Document workflows using flowcharts
  5. Record key decisions from meetings

Deliverables: Interview notes, workflow diagrams, decision log

Assignment 3: Requirement Documentation

Scenario: Document requirements for the library management system.

Tasks:

  1. Create a requirement specification document
  2. Categorize requirements as functional or non-functional
  3. Suggest at least 3 business process improvements
  4. Follow organizational standards for documentation

Deliverables: Requirement specification document with categorized requirements

Assignment 4: Requirement Presentation and Prototyping

Scenario: Present requirements and develop prototypes.

Tasks:

  1. Create a presentation for requirement specifications
  2. Develop low-fidelity prototypes for key features
  3. Present requirements to peers (acting as clients)
  4. Obtain feedback and revise
  5. Create a client confirmation document

Deliverables: Presentation slides, prototypes, confirmation document

Assignment 5: Feasibility Study

Scenario: Conduct a feasibility study for the library management system.

Tasks:

  1. Assess technical feasibility
  2. Conduct economic feasibility analysis
  3. Evaluate operational feasibility
  4. Assess schedule feasibility
  5. Create a comprehensive feasibility study report
  6. Provide recommendations

Deliverables: Complete feasibility study report
