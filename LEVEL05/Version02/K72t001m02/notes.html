System Analysis and Design

K72T001M02 / Level 05 – Teaching Notes & E-book

Relevant Unit: K72T001U02  |  Pre-requisite: K72T001M01

Module Overview

0 Objectives

Module aim: Apply concepts to identify, analyze, prioritize, document, and manage user requirements using requirement gathering and identification techniques.

General Objective

At the end of this module, students will be able to apply system analysis and design concepts to analyze requirements, produce logical designs, document components and data, use standard templates, develop test cases, and prepare the final requirement document.

Specific Objectives

  1. Analyze requirements specified in documentation.
  2. Apply system analysis techniques for logical design.
  3. Identify components and document functionality and data.
  4. Document requirement outcomes according to standard templates.
  5. Develop test cases based on design specifications.
  6. Prepare the final requirement document.

What You Will Learn (In Practice)

Objective 1 – Analyze requirements: You will read requirement documents (e.g. from interviews or workshops), identify what the system must do, and extract clear, testable statements. This means spotting missing or unclear requirements and asking the right follow-up questions.

Objective 2 – Apply analysis techniques: You will use tools such as Data Flow Diagrams (DFD), Use Case diagrams, and ER diagrams to model the system logically before any coding. These diagrams help everyone agree on what the system does and how data moves.

Objective 3 – Identify and document components: You will break the system into parts (processes, data stores, actors, entities) and describe what each part does and what data it uses. This is the basis for design and implementation.

Objective 4 – Document with standard templates: You will write requirement outcomes in a standard format (e.g. numbered requirements, SRS sections) so that designers, developers, and testers can use the same document without confusion.

Objective 5 – Develop test cases: From each requirement or use case, you will write test cases (steps, expected result) so that we can later verify that the built system meets the specification.

Objective 6 – Prepare final requirement document: You will produce one complete SRS that includes introduction, requirements, proposed solution, schedule, and test cases, ready for approval and handover to the development team.

Teaching Material (Weeks 1–14)

1 Introduction to Systems & SDLC

Week 1. Systems, system analysis, and SDLC concepts.

What is a System?

A system is a set of interrelated components that work together to achieve a common goal. It has inputs, processing, outputs, and often feedback.

In more detail: a system takes inputs (e.g. raw data, user actions, materials), applies processing (rules, calculations, workflows), and produces outputs (reports, screens, decisions). Many systems also have feedback: the output or its effect is measured and used to adjust future inputs or processing (e.g. a thermostat adjusts heating based on current temperature).

Examples: an information system (inputs: transactions; process: validate, store, summarize; output: reports, screens); an education system (inputs: students, curriculum; process: teaching, assessment; output: qualified graduates); a banking system (inputs: deposits, withdrawals; process: update accounts, apply rules; output: balances, statements).

Types of Systems

  • Natural systems – e.g. weather, human body.
  • Physical systems – e.g. machines, buildings.
  • Abstract systems – e.g. ideas, plans.
  • Information systems – collect, process, store, and distribute data to support decision-making.

Role of a System Analyst

The system analyst acts as a bridge between business users and technical teams. Business users know what they want the system to do but may not express it in technical terms; developers know how to build software but may not understand the business. The analyst translates needs into clear, structured specifications.

Responsibilities in detail:

  • Gathering and analyzing requirements: Conduct interviews, workshops, and surveys; observe current work; study existing documents. Then filter, clarify, and prioritize what the system must do (functional requirements) and how well (non-functional requirements).
  • Designing solutions: Produce logical models (e.g. DFDs for processes and data flow, ER diagrams for data, use cases for user actions) and suggest high-level architecture. The analyst focuses on what the system does and how it is structured, not on writing code.
  • Communicating with stakeholders: Present findings, explain diagrams, run review meetings, and resolve conflicts. The analyst must be able to explain technical choices in simple language and capture feedback accurately.
  • Documenting specifications and test cases: Write the SRS (Software/System Requirements Specification), maintain requirement IDs, and derive test cases so that testers can verify each requirement later.

System Development Life Cycle (SDLC)

SDLC is a structured process for building information systems. It ensures that we do not jump straight to coding; we first understand the problem, specify requirements, design the solution, then implement and maintain. This reduces the risk of building the wrong system or missing critical needs.

Main phases explained briefly:

  1. Planning – Define scope (what is in and out), objectives, resources, and timeline. Carry out a feasibility study (technical, economic, operational) to decide if the project should go ahead.
  2. Analysis – Gather requirements from users and stakeholders; document functional and non-functional requirements; produce a requirement specification (e.g. SRS). This module focuses heavily on this phase.
  3. Design – Logical design (what the system does: DFDs, ER, use cases) and physical design (how it is built: technology, database structure, screen layouts).
  4. Implementation – Code the system; perform unit and integration testing; conduct user acceptance testing (UAT); deploy to the live environment and train users.
  5. Maintenance – Fix defects, adapt to changes in the business or technology, and improve performance. Most of the cost of a system over its lifetime is in maintenance.
Review
Define a system. Name three types. What does a system analyst do? List the SDLC phases.

2 System Development Life Cycle (SDLC)

Week 2. SDLC phases in detail and SDLC models.

SDLC Phases in Detail

  • Planning: Define project scope (what will and will not be built), objectives, resources (people, budget, tools), and timeline. The feasibility study answers: Can we build it with our technology? (technical) Is it worth the cost? (economic) Will users accept it? (operational). A go/no-go decision is made at the end of planning.
  • Analysis: Gather requirements using techniques such as interviews, surveys, observation, and workshops. Document functional requirements (what the system must do—e.g. “The system shall allow users to reset password”) and non-functional requirements (quality—e.g. “The system shall respond within 2 seconds”). The main deliverable is the requirement specification (SRS), which may include diagrams (DFD, use cases, ER) to clarify behaviour and data.
  • Design: Logical design describes what the system does without tying it to a specific technology (DFD, ER, use cases). Physical design specifies how it will be built: chosen technology (e.g. web, mobile), database schema, screen layouts, and interfaces. Design documents are the input for implementation.
  • Implementation: Developers build the system according to the design. Unit testing checks individual components; integration testing checks that components work together; user acceptance testing (UAT) confirms that the system meets user needs. After sign-off, the system is deployed (installed, configured) and users are trained.
  • Maintenance: After go-live, the system is maintained: fix defects (bugs), adapt to new regulations or business rules, and improve performance or usability. Maintenance continues until the system is retired or replaced.

SDLC Models

ModelDescriptionWhen to use
WaterfallStrictly sequential: Planning → Analysis → Design → Implementation → Maintenance. Each phase is completed and signed off before the next starts. Documentation is produced at each step.When requirements are clear and stable (e.g. regulatory systems, well-defined processes). Not ideal when requirements may change often.
IterativeRepeated cycles: do a subset of design, then implement and test that subset; then add more in the next cycle. The system grows incrementally. Each iteration delivers a working (but possibly incomplete) version.When you want early feedback and can refine requirements as you learn. Reduces risk by exposing problems early.
AgileShort time-boxed sprints (e.g. 2 weeks). Customer prioritizes a backlog; team delivers working features at the end of each sprint. Scope can change between sprints. Emphasis on collaboration and responding to change.When requirements are expected to change and the customer wants to see progress frequently. Common in product and web development.
Deliverables of the Analysis phase typically include: (1) Software/System Requirements Specification (SRS) document, (2) requirement list with IDs and priorities, (3) diagrams such as context/level-0 DFD, use case diagram, and possibly ER diagram, (4) glossary of terms. These are used by designers and developers in the next phase.

What are the deliverables of the Analysis phase? Compare Waterfall and Agile.

3 Requirement Gathering Techniques

Week 3. How to gather and document requirements.

Why Requirements Matter

Clear, complete requirements reduce rework and help deliver the right solution. Poor or vague requirements lead to scope creep (unplanned features), delays (redo work), and user dissatisfaction (the system does not match what they needed). Many project failures are traced back to inadequate requirement gathering and documentation. Spending time in the analysis phase saves time and cost later.

Requirement Gathering Techniques

  • Interviews – One-to-one or group discussions with users and stakeholders. Use open questions (“How do you currently process orders?”) for depth and closed questions (“Do you need a report daily or weekly?”) for specific facts. Pros: Rich detail, follow-up possible. Cons: Time-consuming; may be biased by who you interview. Prepare a list of questions in advance and take notes or record (with permission).
  • Surveys / Questionnaires – Written or online questions sent to many people. Answers are standardized (e.g. yes/no, scale 1–5, multiple choice). Pros: Reach many people quickly; easy to analyse quantitatively. Cons: Less depth; no immediate follow-up. Use when you need to know how many users do X or prefer Y, or when stakeholders are geographically spread.
  • Observation – Watch users doing their job in the real environment (e.g. at their desk, on the shop floor). You see what they actually do, which may differ from what they say. Pros: Reveals real workflow, bottlenecks, and workarounds. Cons: Requires permission and time; users may behave differently when observed. Combine with a short interview after observation to clarify what you saw.
  • Workshops – Facilitated group sessions with several stakeholders (e.g. users, managers, IT). Activities include brainstorming features, prioritization (e.g. must-have vs nice-to-have), and agreeing on scope. Pros: Quick agreement, shared understanding, reduced misunderstandings. Cons: Dominant participants may overshadow others; need a skilled facilitator.
  • Document analysis – Study existing documents: forms, reports, procedure manuals, policy documents. You understand current process, data items, and business rules. Pros: No need to interrupt users for basic facts. Cons: Documents may be outdated or incomplete. Use together with interviews to validate and fill gaps.
  • Prototyping – Build a simple, clickable version of the system (or key screens) and show it to users for feedback. Pros: Users find it easier to react to something they can see and touch; reveals misunderstandings early. Cons: Users may focus on look and feel rather than completeness of requirements. Start with low-fidelity (sketches, wireframes) before high-fidelity (realistic UI).

Documenting Requirements

Record each requirement in a structured format so that nothing is lost and everyone reads the same thing. A typical format includes: ID (e.g. FR-001), description (clear, testable statement), type (functional or non-functional), priority (e.g. must-have, should-have, could-have), and source (e.g. interview with Manager, workshop 1). Use a standard template or a requirement management tool. Avoid vague words like “user-friendly” unless you define them (e.g. “first-time user can complete order in under 3 minutes”).

Example requirement entry
ID: FR-007 | Type: Functional | Priority: Must-have | Source: Interview, Librarian (12 Jan). Description: The system shall allow a librarian to search for a book by title, author, or ISBN and display availability (available / on loan / reserved).
Activity
Design 5 interview questions for a library system. Then design 5 questionnaire items for the same system.

4 Data Flow Diagrams (DFD)

Week 4. Symbols and drawing context and level-0 DFDs.

What is a DFD?

A Data Flow Diagram (DFD) shows how data moves through a system. It uses processes, data flows, data stores, and external entities.

DFD Symbols

  • Process – Transforms data. Shown as a circle or rounded rectangle. Label: verb + noun (e.g. Validate Order).
  • Data flow – Movement of data. Arrow with label (e.g. Order details, Customer name).
  • Data store – Where data is kept. Two parallel lines or rectangle. Label: noun (e.g. Orders, Customers).
  • External entity – Source or destination outside the system. Rectangle. E.g. Customer, Supplier, Bank.

DFD symbol examples

Data flow
Data store
External entity

Process (circle), Data flow (arrow + label), Data store (parallel lines), External entity (rectangle)

Simple context diagram example

Customer → Order → [Order Processing System] → Confirmation → Warehouse

Context Diagram (Level 0)

The context diagram is the highest-level view: it shows the system as one process in the centre, surrounded by external entities. Arrows show data flows in and out. There are no data stores here because stores are inside the system boundary. How to draw: (1) Write the system name in the centre process. (2) List who sends data in and who receives data out. (3) Draw arrows to/from the centre and label each with the data (e.g. Order, Confirmation, Invoice).

Level-0 DFD

The level-0 DFD explodes the single process into major sub-processes (typically 3 to 7). Show data flows between processes and to/from data stores and external entities. Number processes 1.0, 2.0, 3.0. Each process should do one main function (e.g. 1.0 Validate Order, 2.0 Check Stock). Rules: Every data flow must connect to a process. Process names: verb + noun. Data flow names: noun or noun phrase. When you create level-1, inputs and outputs of the child must match the parent (balancing).

Common mistakes: Forgetting to label a data flow; using vague process names (e.g. "Process data"); showing control flow instead of data; putting data stores in the context diagram.

Rules

  • Every data flow must connect to a process.
  • Process names: verb + noun. Data flow names: noun or noun phrase.
  • Balance: inputs and outputs at child level must match parent.
Review
Draw a context diagram for a simple “Order Processing” system. Then draw a level-0 DFD with at least 3 processes.

5 DFD Leveling & Case Study

Week 5. Level-1 and level-2 DFDs; balancing; full case study.

Leveling

Leveling means breaking one process into smaller sub-processes. For example, process 1.0 in the level-0 DFD becomes a separate diagram (level-1) with processes 1.1, 1.2, 1.3, etc. If one of those is still too complex, you can level again: 1.1 becomes 1.1.1, 1.1.2, 1.1.3 in a level-2 DFD. Stop leveling when each process is simple enough to describe in a few sentences or when you have reached a reasonable level of detail for your audience.

Balancing

Balancing means that the child DFD must have exactly the same inputs and outputs as the parent process. For example, if in level-0 process 2.0 has one input flow “Order” and one output flow “Confirmation”, then the level-1 diagram that expands 2.0 must show “Order” entering the diagram (from an entity or another process) and “Confirmation” leaving it. You cannot add new external inputs or outputs at the child level that were not at the parent. Internal flows between 2.1, 2.2, 2.3 and internal data stores used only inside 2.0 are allowed. Checking balance avoids inconsistencies and missing data.

Balancing example
Parent: Process 3.0 “Create Invoice” has input “Approved Order” and output “Invoice”. Child (level-1 for 3.0): “Approved Order” must enter (e.g. into 3.1); “Invoice” must leave (e.g. from 3.3). Flows between 3.1→3.2→3.3 and a data store “Orders” used only inside this diagram are fine.

Case Study Approach

  1. Read the case and list external entities (who sends/receives data?), main processes (what does the system do?), and data stores (where is data kept?). Underline key nouns (entities, data) and verbs (actions, processes).
  2. Draw the context diagram: one process (system name), entities, and main data flows in/out. Get it reviewed before going deeper.
  3. Draw level-0: Break the system into 3–7 major processes. Show flows between them and to/from entities and data stores. Number processes 1.0, 2.0, …
  4. Choose one or two processes to level (usually the most complex). Draw level-1; ensure every input and output of the parent process appears in the child. Label internal flows clearly.
Assignment
For a given case (e.g. Library, Hospital OPD), produce context, level-0, and one level-1 DFD. Ensure balancing.

6 Use Case Diagrams (UML)

Week 6. UML use cases: actors, use cases, relationships.

What is a Use Case?

A use case is a goal that an actor wants to achieve with the system. The diagram shows actors, use cases, and relationships between them.

Elements

  • Actor – User or external system that interacts with the system. Shown as a stick figure or rectangle. E.g. Customer, Admin, Payment Gateway.
  • Use case – Action or function. Oval. Name: verb phrase (e.g. Place Order, Login, Generate Report).
  • Association – Line between actor and use case (actor participates in that use case).
  • Include – One use case always includes another (e.g. Place Order includes Validate Payment). Arrow from base to included, stereotype «include».
  • Extend – Optional extension (e.g. Login may extend with Reset Password). Arrow from extension to base, stereotype «extend».

Use case diagram symbols

Actor
Use case
Association
Include

Actor (stick figure), Use case (oval), Association (line), Include (arrow + stereotype)

Include vs Extend (Detailed)

«include»: The base use case always includes the included use case. Example: “Place Order” always includes “Validate Payment”—you cannot place an order without validating payment. The arrow goes from the base (Place Order) to the included (Validate Payment). Use include when one use case is a required sub-step.

«extend»: The extending use case is optional; it adds behaviour to the base under certain conditions. Example: “Login” may be extended by “Reset Password” when the user has forgotten the password. The arrow goes from the extending use case (Reset Password) to the base (Login). Use extend when the base is complete on its own but can optionally be extended.

Drawing Tips

(1) Identify actors from requirements: who interacts with the system? (users, other systems). (2) List use cases: what goals can each actor achieve? Use verb phrases (Login, View Balance, Generate Report). (3) Connect actors to use cases with lines (association). (4) Add «include» when a use case always calls another; add «extend» when one use case optionally extends another. (5) Place the system boundary (rectangle) around use cases if you want to show what is inside the system.

Review
Draw a use case diagram for an Online Book Store with at least 2 actors and 5 use cases. Use one «include» relationship.

7 ER / EER Diagrams

Week 7. Entity-Relationship and Enhanced ER modelling.

ER Diagram Concepts

  • Entity – Something we store data about (e.g. Student, Course, Order). Rectangle.
  • Attribute – Property of an entity (e.g. StudentID, Name, DoB). Oval or listed in entity box. Underline key attribute(s).
  • Relationship – How entities are related (e.g. Student enrolls in Course). Diamond or line. Cardinality: 1:1, 1:N, M:N.

ER diagram symbols

Entity
Attribute
Relationship
EER (super/subtype)

Entity (rectangle), Attribute (oval), Relationship (diamond), EER supertype/subtype

Cardinality

Cardinality says how many instances of one entity relate to how many of the other. One-to-One (1:1): e.g. one employee has one staff ID card. One-to-Many (1:N): e.g. one department has many employees; one order has many order lines. Many-to-Many (M:N): e.g. many students enroll in many courses. Show cardinality on the relationship (e.g. 1 next to Department, N next to Employee). In some notations you use 1, N, or * (meaning many).

Attributes: Key, Simple, Composite, Multivalued

Key attribute (primary key): uniquely identifies an entity instance (e.g. StudentID). Underline it. Simple attribute: single value (e.g. Name, Age). Composite attribute: can be split (e.g. Address into Street, City, PostCode). Multivalued attribute: more than one value (e.g. Phone numbers); often moved to a separate entity in normalized design.

EER (Enhanced ER)

EER adds supertype/subtype (inheritance). A supertype is a general entity (e.g. Person); subtypes are special cases (e.g. Student, Staff). Subtypes inherit all attributes of the supertype and may have additional attributes. This avoids repeating common attributes and models is-a relationships. Notation: supertype connected to subtypes via a circle with IS-A or a discriminator attribute (e.g. PersonType: Student | Staff).

Drawing an ER Diagram

  1. Identify entities from requirements.
  2. List attributes and primary keys.
  3. Identify relationships and cardinality.
  4. Add subtypes if some entities are special cases of another.
Review
Draw an ER diagram for a Library (Members, Books, Loans). Show attributes and cardinality. Optionally use EER for Member subtypes (e.g. Student Member, Staff Member).

Pseudocode

Pseudocode is an informal, language-like description of logic. It is not a real programming language but is easy to read and later translate into code. Use keywords such as START, END, READ, PRINT, IF…THEN…ELSE, WHILE, FOR, and use indentation to show blocks (what is inside the IF, inside the loop). Use meaningful variable names (e.g. totalPrice, customerName).

Example. READ price, quantity. total = price * quantity. IF total > 1000 THEN discount = 0.1 * total ELSE discount = 0. END IF. PRINT total - discount.

Conventions: One statement per line; use = for assignment; use indentation for nested blocks; capitalize keywords if you like (IF, THEN, ELSE, WHILE, FOR).

When to Use

Flowcharts are good when you want to show flow and decisions visually (e.g. to explain to a stakeholder or to spot missing branches). Pseudocode is good when you are designing an algorithm and plan to code it next; it is closer to actual code and easier to translate. Both can represent the same logic—choose based on audience and purpose.

In-class task
Draw a flowchart for “Check if a number is even or odd”. Then write the same logic in pseudocode.

9 UI/UX Design Principles

Week 9. User interface and user experience basics.

UI vs UX

UI (User Interface) – What the user sees and touches (buttons, forms, layout). UX (User Experience) – Overall experience: ease of use, clarity, satisfaction.

Key Principles

  • Consistency – Same actions and patterns across screens (e.g. Save always in same place).
  • Feedback – System responds to user actions (messages, loading, success/error).
  • Simplicity – Only necessary elements; avoid clutter.
  • Visibility – Important options and status are visible.
  • Accessibility – Usable by people with different abilities (contrast, labels, keyboard).

Wireframes and Prototypes

A wireframe is a simple sketch of layout: boxes for header, navigation, main content, buttons, and key elements. No colours or real text needed—focus on structure and placement. A prototype can be a clickable version that shows user flow (e.g. click Login → see dashboard). Start with low-fidelity (paper sketch or simple tool like Balsamiq) to get feedback on structure; then move to high-fidelity (realistic look and feel) if needed. Wireframes and prototypes help users and designers agree on the UI before development.

Review
Sketch a wireframe for a login screen. Label main elements and explain one UI principle you applied.

10 System Requirements Specification (SRS) – Introduction

Week 10. Purpose and structure of an SRS document.

What is an SRS?

A Software/System Requirements Specification (SRS) is a document that describes what the system must do (functional requirements), how well it must do it (non-functional requirements), and other constraints. It is the contract between stakeholders and the development team.

Purpose

The SRS serves as (1) a single reference for what the system must do, so that business, design, development, and testing all work from the same document; (2) a way to reduce misunderstandings by stating requirements clearly and in one place; (3) a basis for design (designers use it to produce technical design), testing (test cases are derived from requirements), and acceptance (the customer signs off against the SRS).

Typical SRS Sections

  1. Introduction (purpose, scope, definitions)
  2. Executive summary
  3. Overall description (product perspective, user classes, constraints)
  4. Functional requirements
  5. Non-functional requirements (performance, security, usability)
  6. Proposed solution / design overview
  7. Appendices (diagrams, glossary, test cases)

Sources for SRS

Interviews, workshops, existing documents, DFDs, use cases, ER diagrams, and earlier requirement lists.

Draft preparation
Select a project. Create an SRS outline with section headings. Draft the Introduction and Scope (1–2 paragraphs).

11 SRS Structure (Executive Summary, Requirements)

Week 11. Writing executive summary and requirements sections.

Executive Summary

The executive summary is a short overview (often one page) for managers and decision-makers who may not read the full document. It should state: (1) What the project is (system name, purpose); (2) Main goals (what problem it solves or what opportunity it supports); (3) Key features (3–5 high-level capabilities); (4) Main benefits (e.g. faster processing, fewer errors, better reporting). Write it after you have the rest of the SRS so you can summarize accurately. Use clear, non-technical language.

Structuring Requirements

  • Number requirements (e.g. FR-001, FR-002 for functional; NFR-001 for non-functional).
  • One requirement per statement; use “The system shall…” for functional.
  • Group by feature or module (e.g. User Management, Order Processing).
  • Include diagrams (DFD, use case, ER) in the document and refer to them in the text.

Traceability

Trace requirements to their source (e.g. interview, use case) and later to design and test cases. This helps ensure nothing is missed.

Draft review
Review your draft: Are requirements numbered? Is each statement clear and testable? Do diagrams match the text?

12 Proposed Solution & Schedule Structure

Week 12. Describing the solution and project schedule in the SRS.

Proposed Solution Section

This section describes how the system will be built at a high level. Include: (1) Architecture—e.g. web application, client–server, mobile app; (2) Main technologies—e.g. database (MySQL, PostgreSQL), backend (Java, .NET), frontend (React, Angular), or generic categories if specific tools are not yet chosen; (3) How major requirements will be met—e.g. “User management will use role-based access; orders will be stored in a relational database and validated via business rules.” Refer to your DFDs and use cases where they support the description.

Schedule Structure

Show the project timeline so that stakeholders know when to expect deliverables. Use phases or milestones (e.g. Requirements complete, Design complete, Development complete, Testing complete, Go-live). Present in a simple table (Phase, Start date, End date, Deliverable) or a Gantt chart. Be realistic about time and show dependencies (e.g. design must finish before development starts).

Group Work

In groups, complete the proposed solution and schedule for your project. Peer-review each other’s SRS for clarity and completeness.

Workshop
Add “Proposed Solution” and “Schedule” sections to your SRS. Include at least 4 milestones with target dates.

13 Test Case Development

Week 13. Writing test cases from design and requirements.

What is a Test Case?

A test case describes one test: preconditions, steps to run, expected result, and (when run) actual result. It is used to verify that the system meets requirements.

Test Case Components

  • Test case ID
  • Requirement ID (traceability)
  • Description / objective
  • Preconditions
  • Test steps (numbered)
  • Expected result
  • Actual result (filled during execution)
  • Pass / Fail

Deriving Test Cases from Requirements

For each functional requirement, ask: “What would we do to check this?” Write at least one positive test (normal case, valid input, expected behaviour) and, where useful, negative tests (invalid input, wrong user) and boundary tests (min/max values). Use cases and DFDs help identify scenarios. Trace each test case to a requirement ID for coverage.

Example
Requirement FR-012: The system shall allow login with valid username and password. Test cases: TC-012a (positive: valid credentials → success); TC-012b (negative: wrong password → error message); TC-012c (negative: empty username → validation message).

Where to Put Test Cases in SRS

Either in an appendix “Test Cases” or in a separate test specification document. In the SRS, you can list “Test case IDs” under each requirement for traceability.

Review
Write 3 test cases for one requirement (e.g. “User can login with valid username and password”). Use the standard format.

14 Final SRS Preparation & Presentation

Week 14. Completing the SRS and presenting it (Group Assignment 60%).

Final SRS Checklist

  • All sections complete (intro, executive summary, requirements, proposed solution, schedule, appendices).
  • Requirements numbered and traceable; no duplicates or contradictions.
  • Diagrams included and referred to in text (DFD, use case, ER as applicable).
  • Test cases section or appendix with at least a sample set.
  • Consistent formatting, table of contents, and page numbers.
  • Spell-check and proofread.

Presentation Guidelines

Present key parts in a clear order: (1) project aim and scope; (2) main requirements (summarize; refer to numbering); (3) proposed solution (architecture, technology); (4) one or two sample test cases to show traceability. Use slides (title, bullet points, one or two diagrams); avoid long paragraphs. Keep to time; rehearse. Prepare for questions on requirements (e.g. “Why did you prioritise this?”), design (e.g. “Why this architecture?”), and test cases (e.g. “How do you test this requirement?”).

Group Assignment (60%)

Deliverables: (1) Final SRS document (complete, formatted, with all sections and diagrams); (2) Group presentation (typically 10–15 minutes plus Q&A). Marking typically covers: completeness and clarity of the SRS; quality and correctness of requirements and diagrams (DFD, use case, ER); structure and traceability (numbering, test cases linked to requirements); and presentation delivery (clarity, timekeeping, handling of questions).

Tip. Rehearse the presentation. Assign roles (who presents which section). Have one person handle questions or share the role.
