1Identify Test Needs

Identify Situation of Software System

Understanding the software system's context is crucial for effective testing. This involves analyzing various aspects of the system environment and requirements.

Key Areas to Identify:
  • System Architecture: Understanding the overall structure, components, and their interactions
  • Technology Stack: Programming languages, frameworks, databases, and third-party integrations
  • Business Context: Purpose of the system, target users, and business objectives
  • Operating Environment: Hardware, software, network configurations, and deployment scenarios
  • Regulatory Requirements: Compliance standards, security requirements, and industry regulations
  • Performance Expectations: Expected load, response times, and scalability requirements
Software System Context Analysis

Identify Cause of Software Bug

Understanding the root causes of software bugs helps in preventing them and designing better test cases. Bugs can originate from various sources throughout the software development lifecycle.

Bug Category Common Causes Examples
Requirements Issues Ambiguous, incomplete, or changing requirements Missing validation rules, unclear business logic
Design Flaws Poor architecture, incorrect algorithms Inefficient data structures, wrong design patterns
Coding Errors Syntax errors, logic mistakes, typos Off-by-one errors, null pointer exceptions
Integration Issues Component incompatibility, API mismatches Data format mismatches, version conflicts
Environment Problems Configuration errors, resource limitations Missing dependencies, insufficient memory
Human Factors Miscommunication, time pressure, fatigue Copy-paste errors, missed edge cases
Bug Root Cause Analysis

Identify Roles for Development

Different roles in software development have distinct responsibilities and perspectives on testing. Understanding these roles helps coordinate testing efforts effectively.

Project Manager

Oversees project timeline, resources, and ensures testing aligns with project goals. Responsible for risk management and stakeholder communication.

Developer

Writes code, performs unit testing, fixes bugs, and ensures code quality. Responsible for code reviews and technical documentation.

Test Engineer

Designs test cases, executes tests, reports defects, and validates fixes. Ensures comprehensive test coverage and quality assurance.

Business Analyst

Defines requirements, validates business logic, and ensures system meets business needs. Acts as bridge between business and technical teams.

QA Lead

Manages testing strategy, test planning, and quality metrics. Coordinates testing activities and ensures adherence to quality standards.

DevOps Engineer

Manages test environments, CI/CD pipelines, and automation infrastructure. Ensures testing can be performed in production-like environments.

Implement Maintenance

Maintenance testing ensures that changes, updates, and fixes don't introduce new issues while preserving existing functionality.

Maintenance Testing Types:
  • Corrective Maintenance: Testing fixes for reported bugs
  • Adaptive Maintenance: Testing changes for new environments or platforms
  • Perfective Maintenance: Testing performance improvements and enhancements
  • Preventive Maintenance: Testing refactoring and code improvements

Implement Application

Application implementation testing validates that the software works correctly in its intended environment and meets all specified requirements.

2Identify Basic Test Process

Identify Test Design Activities and Control

Test design activities involve creating test cases, test data, and test procedures. Control mechanisms ensure these activities are properly managed and executed.

Test Design Activities:
  • Analyze requirements and specifications
  • Identify test conditions
  • Design test cases
  • Prepare test data
  • Create test procedures
  • Review and approve test designs
Control Mechanisms:
  • Test design standards and templates
  • Review and approval processes
  • Version control for test artifacts
  • Traceability matrices
  • Change management procedures
  • Quality gates and checkpoints

Perform Analysis and Design

Test analysis and design transform test conditions into detailed test cases and test procedures that can be executed.

Test Analysis and Design Process

Implement Development and Execution

Test development involves creating test scripts and automation, while execution involves running tests and collecting results.

Step 1: Test Script Development

Create automated test scripts using appropriate testing frameworks and tools. Scripts should be maintainable, reusable, and well-documented.

Step 2: Test Environment Preparation

Set up test environments that mirror production conditions. Ensure all dependencies, databases, and external services are configured correctly.

Step 3: Test Execution

Execute test cases according to the test plan. This includes manual testing, automated test runs, and exploratory testing sessions.

Step 4: Result Collection

Collect test results, logs, screenshots, and other evidence. Document pass/fail status, execution time, and any anomalies observed.

Implement Verification

Verification ensures that test activities are performed correctly and that the software meets specified requirements.

Verification Type Purpose Activities
Test Case Verification Ensure test cases are correct and complete Review test cases, check coverage, validate test data
Test Execution Verification Confirm tests were executed properly Verify test logs, check environment setup, validate results
Result Verification Validate test results accuracy Compare expected vs actual, verify defect reports, check metrics
Process Verification Ensure testing process is followed Audit test documentation, review adherence to standards, check traceability

Identify Termination Criteria

Termination criteria define when testing should stop. These criteria help ensure adequate testing without unnecessary delays.

Common Termination Criteria:
  • Test Coverage: Achieved required coverage percentage (e.g., 80% code coverage)
  • Defect Rate: Defect discovery rate falls below threshold
  • Time/Budget: Allocated time or budget exhausted
  • Quality Metrics: Quality metrics meet acceptance criteria
  • Risk Assessment: Residual risks are acceptable
  • Stakeholder Approval: Stakeholders approve release readiness

Perform Termination Task

Termination tasks ensure proper closure of testing activities with complete documentation and knowledge transfer.

Test Termination Activities
Key Termination Tasks:
  • Compile final test summary report
  • Document lessons learned and best practices
  • Archive test artifacts and documentation
  • Conduct test closure meeting with stakeholders
  • Update test metrics and dashboards
  • Hand over knowledge to maintenance team

3Identify Software Test Life Cycle

Identify Test Model

Test models provide frameworks for organizing and executing testing activities throughout the software development lifecycle.

V-Model

Sequential model where testing phases correspond to development phases. Each development phase has a corresponding testing phase.

  • Unit Testing ↔ Unit Design
  • Integration Testing ↔ System Design
  • System Testing ↔ Requirements Analysis
  • Acceptance Testing ↔ User Requirements

Agile/Iterative Model

Testing is integrated into each iteration/sprint. Continuous testing with frequent feedback loops.

  • Test-Driven Development (TDD)
  • Behavior-Driven Development (BDD)
  • Continuous Integration/Testing
  • Sprint-based testing cycles

Spiral Model

Risk-driven model with iterative cycles. Each cycle includes risk analysis, testing, and evaluation.

Waterfall Model

Traditional sequential model where testing occurs after development phases are complete.

V-Model Testing Framework

Identify Test Variety

Different types of testing address different aspects of software quality. Understanding test variety ensures comprehensive coverage.

Test Type Purpose Focus Area
Functional Testing Verify system functions correctly Requirements, features, business logic
Non-Functional Testing Verify system quality attributes Performance, security, usability, reliability
Structural Testing Test internal structure Code coverage, paths, branches
Regression Testing Ensure changes don't break existing functionality Previously working features
Smoke Testing Quick validation of basic functionality Critical paths, build stability
Sanity Testing Verify specific functionality after changes Modified areas, related features
Exploratory Testing Discover defects through exploration Unscripted testing, edge cases
Ad-hoc Testing Informal testing without documentation Random scenarios, creative testing

Identify Test Level: Unit Test, Combined Test

Testing is performed at different levels, each focusing on different scopes and objectives.

Test Levels Pyramid

Unit Testing

Definition: Testing individual components or modules in isolation.

  • Performed by developers
  • Tests smallest testable parts
  • Uses mocks and stubs
  • Fast execution
  • High coverage possible

Example: Testing a single function that calculates tax.

Integration Testing (Combined Test)

Definition: Testing combined components/modules to verify interactions.

  • Tests interfaces between components
  • Verifies data flow
  • Checks API integrations
  • Validates system behavior
  • Can be incremental or big-bang

Types: Top-down, Bottom-up, Sandwich, Big-bang

4Test Design Technique

Implement Determination of Test Condition

Test conditions are testable aspects of a component or system identified from requirements, specifications, or other sources. They form the basis for test case design.

Steps to Determine Test Conditions:
  1. Analyze Requirements: Review functional and non-functional requirements
  2. Identify Testable Items: Break down requirements into testable elements
  3. Consider Test Types: Functional, performance, security, usability, etc.
  4. Prioritize: Rank test conditions based on risk and importance
  5. Document: Record test conditions in test design specifications
Test Condition Identification Process
Example Test Condition: Requirement: "User login must validate credentials" Test Conditions: - Valid username and password combination - Invalid username - Invalid password - Empty username field - Empty password field - SQL injection attempt - Session timeout handling

Perform Black Box Test

Black box testing focuses on testing software functionality without knowledge of internal code structure. Tests are based on requirements and specifications.

Characteristics

  • No knowledge of internal structure
  • Tests based on requirements
  • User perspective testing
  • Focus on inputs and outputs
  • Can be performed by non-developers

Advantages

  • Tests from user's viewpoint
  • No programming knowledge needed
  • Unbiased testing
  • Tests actual requirements
  • Early test case design possible

Black Box Testing Techniques:

Technique Description Use Case
Equivalence Partitioning Divide input data into equivalent groups that should produce similar outputs Age validation (0-17, 18-65, 66+)
Boundary Value Analysis Test values at boundaries of input ranges Testing min, max, min-1, max+1 values
Decision Table Testing Test combinations of conditions and actions Complex business rules with multiple conditions
State Transition Testing Test state changes and transitions Login states, order status changes
Use Case Testing Test scenarios based on use cases End-to-end user workflows
Error Guessing Use experience to guess potential errors Common mistakes, edge cases
Black Box Testing Approach
Example: Boundary Value Analysis Input Field: Age (valid range: 18-65) Test Cases: - Min-1: 17 (should fail) - Min: 18 (should pass) - Min+1: 19 (should pass) - Nominal: 40 (should pass) - Max-1: 64 (should pass) - Max: 65 (should pass) - Max+1: 66 (should fail)

Perform White Box Test

White box testing (also called structural or glass box testing) examines the internal structure, logic, and code of the software. It requires knowledge of programming.

Characteristics

  • Requires code knowledge
  • Tests internal structure
  • Focus on code paths
  • Developer perspective
  • Measures code coverage

Coverage Types

  • Statement Coverage: All statements executed
  • Branch Coverage: All branches tested
  • Path Coverage: All paths executed
  • Condition Coverage: All conditions tested

White Box Testing Techniques:

Technique Description Coverage Metric
Statement Coverage Execute every statement at least once Lines of code executed
Branch Coverage Test all true/false outcomes of decisions Decision outcomes tested
Path Coverage Test all possible execution paths Unique paths executed
Condition Coverage Test all boolean expressions Condition outcomes tested
Loop Testing Test loop boundaries and iterations Loop execution scenarios
Data Flow Testing Test variable definitions and uses Definition-use pairs
