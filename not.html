1 Contractual Scope

What is Contractual Scope?

Contractual scope defines the boundaries of work in a contract. It specifies what is included and excluded, guiding all solution planning activities. Think of it as a fence that clearly marks what you must deliver and what is outside your responsibility.

Key Characteristics:

  • Defines boundaries: Clearly states what work is included
  • Specifies exclusions: Explicitly lists what is NOT included
  • Guides planning: All solution activities must align with scope
  • Legal protection: Protects both client and vendor

Real-World Example:

Scenario: A company contracts to develop an e-commerce website.

In Scope: Product catalog, shopping cart, payment processing, user accounts

Out of Scope: Mobile app development, inventory management system, customer support chat

Presenter Notes: Explain inclusions, exclusions, and why scope is legally important. Use the example to illustrate how scope prevents misunderstandings and disputes.

2 Contractual Documents

Contractual documents contain requirements, constraints, and deliverables. They serve as the foundation for solution design and must be carefully analyzed before planning any solution.

RFP (Request for Proposal)

Document requesting vendors to submit proposals for a project. Contains initial requirements and evaluation criteria.

SOW (Statement of Work)

Detailed description of work to be performed, deliverables, timelines, and acceptance criteria.

SLA (Service Level Agreement)

Defines performance metrics, availability requirements, and service quality standards.

Agreements

Legal contracts specifying terms, conditions, payment, and responsibilities of all parties.

Document Hierarchy:

  1. RFP → Client requests solutions
  2. Proposal → Vendor responds with solution
  3. SOW → Detailed work description
  4. Contract → Legal agreement signed
  5. SLA → Ongoing service standards
Presenter Notes: Show or describe real contract documents used in ICT companies. Explain how each document type serves a different purpose in the project lifecycle.

3 Scope Identification

Identifying scope involves extracting all requirements, constraints, and deliverables from contractual documents. This process ensures nothing is missed and nothing extra is included.

Functional Requirements

What the system must DO:

  • User authentication
  • Data processing
  • Report generation
  • Business logic implementation

Non-Functional Requirements

How the system must PERFORM:

  • Performance (speed, response time)
  • Security (encryption, access control)
  • Scalability (handle growth)
  • Usability (user-friendly interface)

Additional Scope Components:

  • Deliverables: Tangible outputs (software, documentation, training)
  • Timelines: Milestones and deadlines
  • Constraints: Budget limits, technology restrictions, compliance requirements
  • Assumptions: Conditions believed to be true (e.g., client will provide data)

Example: E-Learning Platform

Functional: Course enrollment, video playback, quiz system, certificate generation

Non-Functional: Support 1000 concurrent users, 99.9% uptime, SSL encryption, mobile responsive

Deliverables: Web application, admin panel, user documentation, training sessions

Timeline: 6 months with monthly milestones

Constraints: Must use cloud hosting, comply with GDPR, budget: $200,000

Presenter Notes: Explain difference between functional and non-functional requirements with examples. Use the e-learning platform example to show how all components work together.

4 Solution Range

What is Solution Range?

Solution range is a set of possible solutions within the defined scope. These solutions must fully comply with contractual requirements and should avoid out-of-scope features. The goal is to identify multiple viable approaches and select the best one.

Solution Range Characteristics:

  • Within scope: All solutions must meet contractual requirements
  • Feasible: Technically and financially achievable
  • Realistic: Can be delivered within timeline and budget
  • Compliant: Follows all standards and regulations

Example: Database Solution Range

Requirement: Store customer data with high availability

Solution Options:

  1. Option A: MySQL on-premise server (Cost: $50K, Setup: 2 weeks)
  2. Option B: PostgreSQL cloud-hosted (Cost: $30K, Setup: 1 week)
  3. Option C: MongoDB NoSQL solution (Cost: $40K, Setup: 1.5 weeks)

Best Choice: Option B - balances cost, performance, and setup time

Common Mistake:

Proposing solutions with features not in the contract (e.g., adding AI features when not requested) leads to scope creep and budget overruns.

Presenter Notes: Explain solution alternatives using software examples. Emphasize that all options must be within scope and feasible.

5 Solution Range Outlining

Outlining solution range is a systematic process that ensures all contractual requirements are met while identifying the best possible approaches.

Step-by-Step Process:

  1. Analyze the contractual scope
    • Read all contract documents thoroughly
    • Identify all requirements and constraints
    • Note any ambiguities and seek clarification
  2. List mandatory requirements
    • Must-have features and functionalities
    • Non-negotiable constraints
    • Critical deliverables
  3. Identify optional requirements
    • Nice-to-have features
    • Enhancements that add value
    • Future-proofing considerations
  4. Propose feasible and realistic solutions
    • Evaluate technical feasibility
    • Assess cost and timeline
    • Consider risk factors

Example: CRM System Outline

Step 1 - Analysis: Contract requires customer management, sales tracking, reporting

Step 2 - Mandatory: Contact database, sales pipeline, basic reports, user authentication

Step 3 - Optional: Email integration, mobile app, advanced analytics, API access

Step 4 - Solutions:

  • Solution A: Custom-built CRM (6 months, $150K)
  • Solution B: Configured Salesforce (3 months, $80K)
  • Solution C: Open-source SuiteCRM (4 months, $60K)
Presenter Notes: Walk through the process step-by-step. Use the CRM example to show how to systematically analyze and outline solutions.

6 Organizational Standards

What are Organizational Standards?

Organizational standards are company rules and documentation guidelines that ensure consistency and quality across all projects. They include templates, formats, and approval processes that every team member must follow.

Templates

Standardized document formats for proposals, SOWs, reports, and presentations

Guidelines

Writing standards, naming conventions, coding standards, and best practices

Approval Processes

Workflow for document review, sign-offs, and quality checks

Quality Standards

Minimum quality requirements, testing standards, and compliance rules

Why Standards Matter:

  • Consistency: All documents look professional and uniform
  • Quality Assurance: Reduces errors and improves accuracy
  • Efficiency: Saves time with ready-to-use templates
  • Compliance: Meets industry and regulatory requirements
  • Brand Image: Projects professional company image

Example: Company Documentation Standards

  • Template: All proposals use "Company_Proposal_Template_v2.1.docx"
  • Format: Arial 11pt, 1.5 line spacing, page numbers, company logo
  • Structure: Executive Summary → Requirements → Solution → Timeline → Budget
  • Approval: Team Lead → Department Head → Client (3-step process)
  • Version Control: All documents tracked in SharePoint with version numbers
Presenter Notes: Explain why organizations enforce standards. Relate to audits, quality checks, and professional image. Show examples of company templates if available.

7 Applying Standards to Solutions

When creating solution documents, you must use standard templates, follow defined structure and format, and produce professional solution documents that meet organizational quality standards.

Application Process:

  1. Use Standard Templates
    • Download latest version from company repository
    • Ensure template is approved and current
    • Follow template structure exactly
  2. Follow Defined Structure
    • Use required sections in correct order
    • Include all mandatory elements
    • Maintain consistent formatting
  3. Produce Professional Documents
    • Proofread for errors
    • Ensure visual consistency
    • Include proper branding

Example: Solution Document Structure

Standard Template Sections:

  1. Cover Page (with logo, project name, date)
  2. Executive Summary (1-2 pages)
  3. Table of Contents
  4. Project Overview
  5. Requirements Analysis
  6. Proposed Solution
  7. Technical Architecture
  8. Implementation Timeline
  9. Budget Breakdown
  10. Risk Assessment
  11. Appendices

Common Mistakes:

  • Using outdated templates
  • Skipping required sections
  • Inconsistent formatting
  • Missing approval signatures

9 Common Mistakes to Avoid

Understanding common mistakes helps prevent project failures, budget overruns, and client dissatisfaction. Learn from these errors to improve your solution planning.

Critical Mistakes:

  1. Including Out-of-Scope Features
    • Adding features not in contract
    • Result: Scope creep, budget overruns, timeline delays
    • Solution: Always refer back to contract documents
  2. Ignoring Contractual Constraints
    • Proposing solutions that violate budget, timeline, or technology limits
    • Result: Proposal rejection, contract breach
    • Solution: List all constraints and verify compliance
  3. Not Following Organizational Standards
    • Using wrong templates, skipping sections, poor formatting
    • Result: Document rejection, rework, professional image damage
    • Solution: Use standard templates and follow guidelines
  4. Incomplete Requirements Analysis
    • Missing requirements or misunderstanding scope
    • Result: Incomplete solutions, client dissatisfaction
    • Solution: Thorough analysis and clarification meetings
  5. Unrealistic Solutions
    • Proposing solutions that can't be delivered
    • Result: Project failure, reputation damage
    • Solution: Honest feasibility assessment

Real Example: Failed Project

Mistake: Developer added mobile app feature not in contract

Impact: - Extra $50K cost not approved
- 2-month delay
- Client refused to pay extra
- Project went over budget by 25%

Lesson: Always stick to contractual scope. Propose extras separately as change requests.
